# xQuakShell Plugin Author Guide

This document describes how to build, pack, sign, and install plugins for xQuakShell.

## Prerequisites

- Go 1.25+
- `xqs-plugin` CLI (build with `make plugin-cli`)
- `CGO_ENABLED=0` for portable static plugin binaries on Windows

## Project layout

```
my-plugin/
  plugin.json       # manifest (required)
  my-plugin.exe     # go-binary entry (built artifact)
  ui/               # optional WebView assets
  data/             # runtime data (created by core)
```

## Quick start

```bash
xqs-plugin init -id com.example.myplugin -name "My Plugin"
xqs-plugin build
xqs-plugin validate
xqs-plugin pack -o dist/my-plugin.xqs-plugin
```

## Manifest

See [plugin-manifest.md](./plugin-manifest.md) for the full schema.

Required fields: `id`, `name`, `version`, `engine.type` (`go-binary`), `engine.entry`.

## Bundles (`.xqs-plugin`)

A bundle is a ZIP archive containing:

- `plugin.json`
- plugin binary and assets
- `SHA256SUMS` (generated by `xqs-plugin pack`)

Install via **Settings → Plugins → Install folder…** or **Install bundle…** (`.xqs-plugin` file).

Installed plugins are copied to `data/plugins/<id>/` under the xQuakShell executable directory (ADR-006 portable layout).

## Security limits

- **IPC frames:** NDJSON lines are capped at **256 KiB**; oversize frames are rejected.
- **Filesystem reads:** `fs.read` returns at most **256 KiB** per call; use `offset` to read larger files (max **16 MiB** total per file).
- **Filesystem writes:** `fs.write` accepts at most **256 KiB** per chunk; omit `offset` to replace a file, or set `offset` for chunked writes.
- **Network:** outbound patterns must be explicit `tcp:hostname:port` — wildcards in the host are rejected. Dial failures are reported as capability denied (no host leakage).
- **Vault IDOR:** plugins may access a connection only while they own an active session for that connection.
- **Secrets:** `vault.getSecret` requires install-time user consent plus manifest declaration. Passphrases require an encrypted identity and an unlocked session cache.
- **Manual start:** `StartPluginManual` requires `onManual` in `activationEvents`.
- **Plugin filesystem:** `pluginsdk` `fs.*` calls remain manifest-sandboxed. The host Local Files browser uses `HostFileSystem` (ADR-007) and is not available to plugins.

See [security-model.md](./security-model.md) for the full threat model.

## Plugin SDK

Use the published module in standalone plugin projects:

```go
import "github.com/xquakshell/pluginsdk"
```

For in-repo development, the core `go.mod` includes:

```
replace github.com/xquakshell/pluginsdk => ./pkg/pluginsdk
```

The typed `pluginsdk.Client` wraps `Host.CallCore` / `CallCoreContext` with a **5s default timeout** and helpers for vault, fs, and events.

Use `Host.OnDeactivate` (notification) and `Host.OnShutdown` (RPC request) for graceful teardown.

## Signing (optional)

1. Generate a publisher key pair:

   ```bash
   xqs-plugin keygen -out publisher.key
   ```

2. Generate checksums and sign the manifest (checksums must exist before signing):

   ```bash
   xqs-plugin checksums -dir .
   xqs-plugin sign -key publisher.key
   xqs-plugin pack -o dist/my-plugin.xqs-plugin
   ```

3. Add `publisher.pub` (base64 Ed25519 public key) to **Settings → Plugins → Trusted publishers** in xQuakShell.

Signatures use Ed25519 over a canonical JSON envelope `{manifest, checksumsSha256}` where `checksumsSha256` is the SHA-256 (hex) of normalized `SHA256SUMS` bytes. Legacy manifest-only signatures must be re-signed after binding checksums.

## IPC overview

Plugins run out-of-process and communicate via JSON-RPC over NDJSON on stdin/stdout. The host calls methods like `initialize`, `activate`, `ping`, and plugin-contributed handlers.

**Shutdown sequence (host → plugin):**

1. Notification `deactivate` (no response)
2. RPC `shutdown` (plugin returns `{"ok":true}`; 2s timeout)
3. Stdin closed; process exit or force-kill after grace period

Session plugins must declare every contributed protocol in `capabilities.session.connectProtocols`.

Capabilities declared in `plugin.json` are enforced by the core capability gate.

## Cross-compilation

```bash
GOOS=windows GOARCH=amd64 CGO_ENABLED=0 go build -ldflags="-s -w" -trimpath -o my-plugin.exe .
```

The core validates the binary matches the host OS at install time.

## Reference plugins

- `plugins/example-echo` — commands, views, status bar
- `plugins/demo-terminal` — session connector demo
- `plugins/example-events` — event bus subscribe/publish

## UI integration

### Wails methods

- `StartPlugin(pluginId)` — manual start (audit-logged, respects disabled flag, idempotent if already running)
- `SetPluginEnabled(pluginId, enabled)` — toggle plugin; disabling stops the process
- `PingPlugin(pluginId)` — ping RPC (does not auto-start)

### Frontend events

- `PluginContributionsChanged` — refresh merged commands/views/status bar
- `PluginStateChanged` — `{ pluginId, state, sessionId? }` where state is `starting|running|stopped|suspended|crashed`
- `PluginViewMessage` — plugin → host view relay
